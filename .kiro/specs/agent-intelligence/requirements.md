# Requirements Document

## Introduction

This document defines the formal requirements for the Agent Intelligence feature set in Tachy, an enterprise on-prem AI agent platform built in Rust. The intelligence layer adds five composable features — Codebase Indexing, Plan-and-Execute Loop, Edit-Test-Fix Cycle, Git Integration, and Smart Context Selection — that elevate Tachy from a bare tool-use loop to a competitive coding agent. Each feature can operate independently but compounds with the others. All features live in a new `intelligence` crate with one-way dependencies on existing crates (`runtime`, `tools`, `backend`).

## Glossary

- **Indexer**: The `CodebaseIndexer` component that scans a workspace directory, extracts file metadata (language, exports, summary, content hash), and persists a searchable `CodebaseIndex` to `.tachy/index.json`.
- **CodebaseIndex**: A serializable data structure (`BTreeMap<String, FileEntry>`) containing metadata for all indexed source files in a workspace, plus project-level metadata.
- **FileEntry**: A single indexed file record containing path, language, size, line count, exports, summary, and content hash.
- **PlanExecutor**: The component that wraps `ConversationRuntime` in an outer planning loop, generating a multi-step `Plan` from a user prompt and executing steps sequentially.
- **Plan**: An ordered sequence of `PlanStep` objects generated by the LLM, each with a description, instruction, expected files, and execution status.
- **PlanStep**: A single atomic unit of work within a Plan, containing an instruction to send to `ConversationRuntime::run_turn()`.
- **EditTestFix**: The component that runs the project's test suite after code edits, and if tests fail, feeds the failure output back to the LLM for automated fix attempts.
- **CycleResult**: The outcome of an EditTestFix cycle: Passed, Fixed, Failed, NoTestCommand, or Timeout.
- **GitTools**: The component providing structured git operations (status, diff, branch, commit) via shell commands, exposed as agent tools.
- **ContextSelector**: The component that analyzes a user prompt, queries the CodebaseIndex for relevant files, and injects file summaries and contents into the system prompt within a token budget.
- **ContextInjection**: The rendered context payload (summaries + file contents) prepended to the system prompt before each `run_turn()`.
- **IntelligenceConfig**: The unified configuration struct controlling which intelligence features are enabled and their parameters.
- **ConversationRuntime**: The existing agentic tool-use loop in the `runtime` crate that processes user turns, calls the LLM, and executes tools.
- **AgentEngine**: The existing orchestrator in the `daemon` crate that creates backends, builds runtimes, and runs agents.
- **GovernancePolicy**: The existing audit and policy enforcement layer that controls which tools an agent can use.

## Requirements

### Requirement 1: Codebase Index Construction

**User Story:** As a developer, I want Tachy to build a searchable index of my project's source files, so that the agent understands the codebase structure without reading every file on each interaction.

#### Acceptance Criteria

1. WHEN `build_index` is called with a valid workspace root, THE Indexer SHALL scan all source files and return a CodebaseIndex where `files.len()` equals `project.total_files`
2. WHEN the Indexer encounters a file, THE Indexer SHALL detect the programming language from the file extension using a deterministic mapping (e.g., `.rs` → `rust`, `.py` → `python`, `.ts` → `typescript`)
3. WHEN the Indexer processes a source file, THE Indexer SHALL extract up to 20 exported symbol names (function names, struct names, class names) using language-specific regex patterns
4. WHEN the Indexer processes a source file, THE Indexer SHALL generate a one-line summary truncated to 120 characters from the first doc comment or first non-empty line
5. WHEN the Indexer encounters files in ignored directories (`.git/`, `.tachy/`, `node_modules/`, `target/`, `__pycache__/`, `.venv/`, `vendor/`), THE Indexer SHALL skip those files
6. WHEN the Indexer encounters a binary file (detected by extension: `.png`, `.jpg`, `.wasm`, `.o`, `.so`, `.exe`), THE Indexer SHALL skip that file
7. WHEN the Indexer encounters a file larger than the configured `max_file_size` (default 100KB), THE Indexer SHALL skip that file
8. WHEN the Indexer encounters files matching patterns in `.gitignore`, THE Indexer SHALL skip those files
9. WHEN the Indexer scans the workspace root, THE Indexer SHALL detect the project's test command by checking for build files (`Cargo.toml` → `cargo test`, `package.json` → `npm test`, `pyproject.toml` → `pytest`, `go.mod` → `go test ./...`, `Makefile` → `make test`)

### Requirement 2: Codebase Index Persistence and Incremental Update

**User Story:** As a developer, I want the codebase index to persist across sessions and update incrementally, so that re-indexing is fast after small changes.

#### Acceptance Criteria

1. WHEN `save_index` is called, THE Indexer SHALL write the CodebaseIndex as pretty-printed JSON to `.tachy/index.json`
2. WHEN `load_index` is called and `.tachy/index.json` exists, THE Indexer SHALL deserialize and return the persisted CodebaseIndex
3. WHEN `save_index` is called followed by `load_index`, THE Indexer SHALL return a CodebaseIndex equivalent to the original (round-trip property)
4. WHEN `update_index` is called with an existing index, THE Indexer SHALL only re-index files whose content hash has changed, and return the count of re-indexed files
5. IF the workspace root passed to `build_index` does not exist, THEN THE Indexer SHALL return an `IndexError::WorkspaceNotFound` error
6. IF `.tachy/index.json` cannot be read or parsed, THEN THE Indexer SHALL return an `IndexError::Io` or `IndexError::Json` error

### Requirement 3: Codebase Index Search

**User Story:** As a developer, I want to search the codebase index by keyword, so that the agent can quickly find relevant files without scanning the filesystem.

#### Acceptance Criteria

1. WHEN `search` is called with a query string, THE Indexer SHALL match against file path, exports, summary, and language fields
2. WHEN `search` is called with a `max_results` parameter, THE Indexer SHALL return at most `max_results` entries
3. WHEN `search` returns results, THE Indexer SHALL sort results by relevance to the query

### Requirement 4: Plan Generation

**User Story:** As a developer, I want Tachy to generate a multi-step plan before executing complex tasks, so that work is broken into atomic, verifiable steps.

#### Acceptance Criteria

1. WHEN `create_plan` is called with a user prompt, THE PlanExecutor SHALL send a structured planning prompt to the LLM and parse the response into a Plan
2. WHEN a Plan is created, THE PlanExecutor SHALL produce between 1 and `max_steps` (default 10) PlanSteps, all in `Pending` status
3. WHEN a Plan is created, each PlanStep SHALL contain a description, an instruction, and a list of expected files
4. IF the LLM fails to generate a valid JSON plan, THEN THE PlanExecutor SHALL return a `PlanError::PlanGeneration` error
5. IF planning is disabled in IntelligenceConfig, THEN THE AgentEngine SHALL fall back to a single `ConversationRuntime::run_turn()` call

### Requirement 5: Plan Execution

**User Story:** As a developer, I want Tachy to execute plan steps sequentially with automatic error recovery, so that complex tasks complete reliably.

#### Acceptance Criteria

1. WHEN `execute` is called on a Plan, THE PlanExecutor SHALL process steps in order, calling `ConversationRuntime::run_turn()` for each step's instruction
2. WHEN a step completes successfully, THE PlanExecutor SHALL mark the step as `Completed` and record a result summary
3. WHEN a step fails, THE PlanExecutor SHALL ask the LLM to revise the remaining plan steps
4. WHEN plan revision occurs, THE PlanExecutor SHALL replace only the remaining unexecuted steps with the revised steps
5. WHEN the number of revisions reaches `max_revisions` (default 3), THE PlanExecutor SHALL stop execution and report partial results
6. WHEN all steps complete, THE PlanExecutor SHALL return a `PlanExecutionResult` containing the final plan, total iterations, total tool invocations, revision count, and success status
7. WHEN `auto_branch` is enabled, THE PlanExecutor SHALL create a git branch named `tachy/task-{id}` before executing the first step
8. WHEN `auto_commit` is enabled and a step succeeds, THE PlanExecutor SHALL commit the changes with the step description as the commit message
9. WHEN `auto_test` is enabled and a step edits code, THE PlanExecutor SHALL trigger an EditTestFix cycle after the step

### Requirement 6: Edit-Test-Fix Cycle

**User Story:** As a developer, I want Tachy to automatically run tests after code edits and fix failures, so that code changes are validated without manual intervention.

#### Acceptance Criteria

1. WHEN `run_cycle` is called with a list of edited files, THE EditTestFix SHALL detect or use the configured test command and execute it
2. WHEN tests pass on the first run (exit code 0), THE EditTestFix SHALL return `CycleResult` with outcome `Passed` and `retries` equal to 0
3. WHEN tests fail, THE EditTestFix SHALL feed the failure output (stderr truncated to 4000 chars, stdout tail 2000 chars) to the LLM with a fix prompt
4. WHEN the LLM produces a fix and tests pass on retry, THE EditTestFix SHALL return `CycleResult` with outcome `Fixed` and the number of retries
5. WHEN tests still fail after `max_retries` (default 3) fix attempts, THE EditTestFix SHALL return `CycleResult` with outcome `Failed`
6. IF no test command can be detected or configured, THEN THE EditTestFix SHALL return `CycleResult` with outcome `NoTestCommand`
7. IF the test command exceeds `test_timeout_secs` (default 120), THEN THE EditTestFix SHALL return `CycleResult` with outcome `Timeout`
8. WHEN `targeted_tests` is enabled, THE EditTestFix SHALL build a targeted test command that runs only tests related to the edited files (e.g., `cargo test indexer` for `src/indexer.rs`)

### Requirement 7: Git Status and Diff

**User Story:** As a developer, I want Tachy to inspect git status and diffs with structured output, so that the agent understands what has changed before committing.

#### Acceptance Criteria

1. WHEN `git_status` is called, THE GitTools SHALL execute `git status --porcelain` and return a structured `GitStatus` with branch name, staged changes, unstaged changes, and untracked files
2. WHEN `git_status` returns, THE GitTools SHALL set `is_clean` to true if and only if staged, unstaged, and untracked lists are all empty
3. WHEN `git_diff` is called, THE GitTools SHALL execute `git diff` and return a structured `GitDiff` with per-file diffs and summary stats (files changed, insertions, deletions)
4. WHEN `git_diff` is called with `staged: true`, THE GitTools SHALL execute `git diff --staged` to show only staged changes
5. WHEN `git_diff` is called with a specific path, THE GitTools SHALL scope the diff to that path
6. IF the current directory is not a git repository, THEN THE GitTools SHALL return a `GitError::NotARepo` error

### Requirement 8: Git Branch and Commit

**User Story:** As a developer, I want Tachy to create branches and commit changes programmatically, so that agent work is safely isolated and versioned.

#### Acceptance Criteria

1. WHEN `git_branch` is called with `create: true`, THE GitTools SHALL execute `git checkout -b {name}` and return a `BranchResult` with `created: true` and the previous branch name
2. WHEN `git_branch` is called with `create: false`, THE GitTools SHALL switch to the existing branch
3. WHEN `git_commit` is called with a message, THE GitTools SHALL execute `git add -A && git commit -m "{message}"` and return a `CommitResult` with the commit hash (7+ characters), message, and files changed count
4. WHEN the PlanExecutor creates a branch, THE GitTools SHALL prefix the branch name with `tachy/` to distinguish agent-created branches
5. IF there are no changes to commit, THEN THE GitTools SHALL return a `GitError::NothingToCommit` error
6. WHEN a git commit is made, THE GovernancePolicy SHALL log an audit event for the commit

### Requirement 9: Git Tool Registration

**User Story:** As a developer, I want git operations exposed as agent tools, so that the LLM can use them during task execution.

#### Acceptance Criteria

1. THE AgentEngine SHALL register `git_status`, `git_diff`, `git_branch`, and `git_commit` as available tools with proper JSON schemas
2. WHEN the GovernancePolicy's `allowed_tools` list does not include a git tool, THE AgentEngine SHALL block that git tool from execution
3. WHEN a git tool is invoked, THE AgentEngine SHALL delegate execution to the corresponding `GitTools` function

### Requirement 10: Smart Context Selection

**User Story:** As a developer, I want Tachy to automatically inject relevant file contents into the model's context based on my prompt, so that the local model has the information it needs without exceeding its context window.

#### Acceptance Criteria

1. WHEN `select_context` is called with a prompt and a CodebaseIndex, THE ContextSelector SHALL extract keywords (file paths, function names, module names, error messages) from the prompt
2. WHEN the ContextSelector ranks files, THE ContextSelector SHALL score each file based on: direct path mention in prompt (highest weight), filename match, export name match, summary keyword overlap, and penalize files over 500 lines
3. WHEN the ContextSelector builds a ContextInjection, THE ContextSelector SHALL include at most `max_full_files` (default 5) full file contents and at most `max_summaries` (default 20) file summaries
4. WHEN the ContextSelector builds a ContextInjection, THE ContextSelector SHALL ensure `estimated_tokens` does not exceed `max_context_percentage` (default 40%) of the model's context window
5. WHEN the ContextSelector orders file contents in the injection, THE ContextSelector SHALL order them by relevance score (highest first)
6. WHEN the ContextSelector renders the injection, THE ContextSelector SHALL produce a formatted string containing project metadata, file summaries with exports, and file contents with language annotations
7. WHEN context selection is enabled, THE AgentEngine SHALL call `select_context` before each `run_turn()` and prepend the rendered injection to the system prompt

### Requirement 11: Intelligence Configuration

**User Story:** As a developer, I want to enable or disable each intelligence feature independently via configuration, so that I can tune Tachy's behavior for my hardware and workflow.

#### Acceptance Criteria

1. THE IntelligenceConfig SHALL provide boolean flags for each feature: `indexing_enabled`, `context_enabled`, `planning_enabled`, `edit_test_fix_enabled`, `git_enabled`
2. WHEN a feature flag is set to false, THE AgentEngine SHALL skip that feature and fall back to existing behavior
3. THE IntelligenceConfig SHALL be loadable from `.tachy/config.json` under the `intelligence` key
4. THE IntelligenceConfig SHALL provide sensible defaults: all features enabled, `max_steps: 10`, `max_revisions: 3`, `max_retries: 3`, `test_timeout_secs: 120`, `max_context_percentage: 0.40`, `max_full_files: 5`, `max_summaries: 20`

### Requirement 12: Graceful Degradation

**User Story:** As a developer, I want Tachy to continue working even when intelligence features fail, so that a broken index or missing git repo does not block my workflow.

#### Acceptance Criteria

1. IF the Indexer fails with an `IndexError::Io`, THEN THE AgentEngine SHALL log a warning and continue without the codebase index
2. IF the PlanExecutor fails with `PlanError::PlanGeneration`, THEN THE AgentEngine SHALL fall back to a single `run_turn()` without planning
3. IF the PlanExecutor fails with `PlanError::StepFailed` after max revisions, THEN THE PlanExecutor SHALL report partial results for completed steps
4. IF the EditTestFix returns `NoTestCommand`, THEN THE PlanExecutor SHALL skip the ETF cycle and log an informational message
5. IF the EditTestFix returns `Timeout`, THEN THE PlanExecutor SHALL report the timeout and continue to the next plan step
6. IF the GitTools returns `NotARepo`, THEN THE AgentEngine SHALL skip all git operations and log an informational message
7. IF the ContextSelector fails with `NoIndex`, THEN THE AgentEngine SHALL use the base system prompt without context injection

### Requirement 13: Intelligence Crate Architecture

**User Story:** As a developer, I want all intelligence features in a single crate with one-way dependencies, so that the codebase stays modular and existing crates are not modified structurally.

#### Acceptance Criteria

1. THE Intelligence crate SHALL contain modules: `indexer.rs`, `planner.rs`, `edit_test_fix.rs`, `git.rs`, `context.rs`, and `lib.rs`
2. THE Intelligence crate SHALL depend on `runtime`, `tools`, and `backend` crates
3. THE Intelligence crate SHALL NOT be depended upon by `runtime`, `tools`, or `backend` (one-way dependency flow)
4. WHEN the `daemon` or `tachy-cli` integrates intelligence features, THE integration SHALL wire dependencies at startup without circular crate dependencies

### Requirement 14: Index Serialization Format

**User Story:** As a developer, I want the codebase index stored as human-readable JSON with a schema version, so that I can inspect it and future versions can migrate the format.

#### Acceptance Criteria

1. THE CodebaseIndex SHALL include a `version` field (starting at 1) for forward compatibility
2. THE CodebaseIndex SHALL include a `workspace_root` field recording the absolute path of the indexed workspace
3. THE CodebaseIndex SHALL include a `built_at` field recording the unix timestamp of the last build
4. THE CodebaseIndex SHALL store files in a `BTreeMap` keyed by relative path for deterministic ordering
5. WHEN the Indexer serializes the index, THE Indexer SHALL use pretty-printed JSON format
